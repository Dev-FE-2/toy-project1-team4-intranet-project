#### 마이페이지

- 실시간으로 시간을 표시한다.
- 사용자 프로필 정보가 나타난다.
- 근무 시작 및 종료 버튼 클릭으로 모달창을 호출한다.
- 호출된 모달창에서 버튼을 통해 근무 시작과 근무 종료를 수행한다.
- 이번 주 근무시간이 도넛 차트와 시간으로 반영된다.
- 근태 목록 프리뷰가 제공된다.

#### 사용자 프로필 페이지
- 프로필 사진 등록/수정/삭제 가능하다.
- 프로필 정보 수정할 수 있다.
#### 직원 프로필 등록 페이지
- 직원 프로필 사진 등록/수정/삭제 가능하다.
- 새로운 직원 프로필 정보 등록할 수 있다.
#### 직원 프로필 상세 페이지
- 직원 프로필 사진 등록/수정/삭제 가능하다.
- 직원 프로필 정보를 수정할 수 있다.
- 직원 삭제 버튼으로 삭제할 수 있다.
#### 직원 목록 페이지
- 총 직원의 수를 확인 가능하다.
- 이름으로 검색해서 직원 조회할 수 있다.
- 직원 등록 버튼으로 직원 등록 페이지로 이동한다.
- 직원 목록은 10개씩 보여지고,  스크롤로 확인 가능하다.
- 목록 스크롤 하단에 페이지네이션으로 다음 목록을 조회할 수 있다.
- 목록에서 특정 직원 클릭 시 직원 프로필 상세 페이지로 이동한다.

#### 근태 신청 페이지
- 근태를 신청한다.
- 근태 신청 모든 내역을 조회한다.
- 근태 신청 상세 내역을 확인한다.
- 나의 근태 내역만 필터링 하여 조회한다.
- 연차, 반차, 조퇴, 기타 메뉴 별로 필터링하여 조회한다.
- 한 페이지에 10개씩 조회한다,
- 페이지네이션이 동작한다,

#### 공지사항 페이지
- 공지사항 내역을 조회한다.
- 목록 스크롤 하단에 페이지네이션으로 다음 목록을 조회할 수 있다.
- 레이지로딩(MO) 기능을 사용한다.
- 공지사항 내역 클릭시 상세페이지로 이동한다.
- 목록에서 특정 공지사항을 클릭 시 공지사항 상세 페이지로 이동한다.

### 브랜치전략

- **main, develop** 브랜치와 **feature** 보조 브랜치를 운용했습니다.
  - **main**: 배포 단계에서만 사용하는 브랜치
  - **develop**: 개발 단계에서 git-flow의 master 역할을 하는 브랜치
  - **Feat**: 기능 단위로 독립적인 개발 환경을 위하여 사용하고 merge 후 각 브랜치를 삭제
    - feature 브랜치 이름 규칙: `feature/기능명-issue번호`

## 협업방식

**🗣️ 슬랙을 사용한 활발한 의사소통**

프로젝트 진행 동안 활발한 의사소통을 유지하며 신속하고 효율적인 업무 처리가 가능했습니다.
 <img width="1054" alt="슬랙" src="https://github.com/user-attachments/assets/16e06982-9e25-4a39-8173-f8845fa10a89">


**📃 노션 데이터베이스를 활용한 기능 명세서와 타임라인**

기능명세서는 팀원과의 협의를 통해 계층형으로 작성하여 각 페이지 별 기능을 명확하게 구분하였고, 
타임라인과의 연동과 원 차트를 통해 쉽게 작업 진행 상황을 파악할 수 있도록 하였습니다.
 <img width="1054" alt="기능 명세서" src="https://github.com/user-attachments/assets/0914ddde-e1f4-4904-ae6e-056175a875fe">

**📒 노션 회의록**

회의록은 노션을 통해 관리하며, 모든 팀원이 각자 역할 분담을 작성합니다.
 <img width="1054" alt="회의록" src="https://github.com/user-attachments/assets/5367a8f2-9f77-45cb-b5b0-8527f95f759d">

**✍️ 피그마를 이용한 디자인**

<img width="1054" alt="피그마" src="https://github.com/user-attachments/assets/e83a6e4d-bfe1-4cf7-b171-1ab9535b5395">


피그마를 사용하여 프로젝트의 디자인을 협업했습니다. 데스크탑과 모바일 뷰, 컴포넌트들을 작업하였으며, 팀원들은 제작된 피그마를 바탕으로 마크업 작업을 수월하게 진행 할 수 있었습니다.

**💬 깃허브를 사용한 Issue 관리와 PR 리뷰**

팀원들은 서로의 코드를 검토하고, 개선할 점을 제안하며 프로젝트의 코드 품질을 높였습니다. 이를 통해 코드 통합 과정에서 발생할 수 있는 문제를 최소화하였습니다.

- 모든 브랜치에 2명 이상이 Approve를 해야 PR이 Merge될 수 있도록 설정하였습니다.
- 컨벤션 뿐만 아니라 로직 및 아키텍처 관점에서의 의견 공유도 활발히 진행하였습니다.
- 코드 리뷰를 꼼꼼히 진행하면서 팀원 모두가 프로젝트의 전반적인 코드를 이해하는데 도움이 되었습니다.
